# Live API Reference

This folder contains curated reference documentation for the Ableton Live Python API.
It is the authoritative source of truth for what Live actually exposes â€” independent of
how any particular client library wraps it.

## Purpose

Ableton does not publicly document the Live Python API. The best available sources are:

1. **API stubs** (`build/<version>/Live/classes/*.py`) â€” generated by running a control surface
   script inside Live. Complete coverage of all classes, properties, and methods, but docstrings
   are sparse, argument types are sometimes weak, and behavioral gotchas are not documented.

2. **Max for Live docs** (`MaxForLive/*.md`) â€” richer descriptions, clearer types, and better
   gotcha coverage. Does not cover the full API and is not always kept in sync with the stubs.

3. **Direct probing** â€” for behavior that neither source clarifies (async visibility, error
   conditions, edge cases), we probe the API directly at runtime and record findings as
   probe notes.

The goal of this folder is to synthesize all three sources into a single clean, complete,
and accurate reference for each Live API class.

## What Belongs Here

- Accurate descriptions of what each member does, including quirks and constraints
- Probe-confirmed behavioral notes (async visibility, error conditions, sentinel values)
- Open questions to probe later
- No client library surface mapping â€” this is Live API only

## Adding a New File

1. Copy the template from `_Format.md`.
2. Use the API stub as the primary source for member inventory.
3. Cross-reference the Max for Live docs for better descriptions and gotchas.
4. Mark `Probe Status` honestly: `unprobed`, `partial`, or `verified`.
5. Record unresolved questions in the `Open Questions` section.
6. Add a link to the new file in this README.

## Policies

- Use `Unknown` for any field that cannot be determined from available sources.
- Do not paste raw stub docstrings verbatim â€” rewrite in plain language.
- When a field is enum-backed, use the enum type name (e.g. `Song.RecordingQuantization`)
  and document known numeric mappings in the description.
- For children returning sequences of Live objects, use `Sequence[Type]`.
- Document the API as seen by internal Live scripts/control-surface code.
